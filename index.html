And at the end they make no sense at all! So with BEM block__element_modifier simple rules we had at least understanding throughout of all Yandex services.

Problems → Solution

  • 0 incapsulation...
  • ...so, BEM helped us incapsulate code on block level
  • and made possible to re-use
Not only from project to project, but between services profiting from the designer's idea of having common Yandex brand UI parts. Now we even have Yandex Lego Library that keeps these parts and a team that maintains it. With every new release services that are built on Yandex.Lego as a core library get pieces of common code tested and delivered and have time to concenrate on features.

Problems → Solution

  • More demand on interactive interfaces...
  • ...was not longer a threat
  • BEM file structure gave an option to separate code according to technologies
Interactive interfaces forced us to use more and more Javascript on the page that made maintainance harder and harder every day with high risks and no change to re-use it. Time of developement and maintainance grew up, developers went crazy trying to cope with it switching from project to project. So, guess how creating a plain file structure in BEM helped us live better life! Now you can simply find a block on the page, go into the BEM tree, find a block folder, find a technology file there (css, js, tests, docs, etc) and do whatever you need to do with it!
Of course, there are side-effects as everywhere. One may say that now we get long css classes now one can write down. But we don't write them with our bare hands. We automated everything we can giving ourselves a chance to think of what we do and how we do that, and not to do something without thinking.

What BEM can do that others can't

BEM is a remedy for all this pain of web development.

Abstraction

  • speak the same terms
  • code becomes self-documented
  • possibility to update components appear
All these divs and spans gives developer no clue about what they really are. Actualy we use just the same low level HTML invented for browsers in 1993. But web applications nowadays are much closer to desktop applications then to static documents of Web 1.0. We need high level abstraction of assembler browsers suggest.

In short

  • Block
  • Elem
  • Modifier
First of all let's define what are blocks, elements and modifiers. A block is an independent entity with its own meaning that represents a piece of interface on a page. It is a component in terms of web components specification. Blocks are intended to be reusable and easily extendable. Element is a part of block which has no meaning without it's parent block. Often elements are private part of API so you can think of them as of shadow DOM. But here you can see that we prefer convention over configuring: technically you can always interact with any element but because of our naming system one can't do it without [осознанного] intention. Modifiers represent differences between similar blocks and — which is more important — the state of a block or an element.

Some of BEM principles

  • Map document to BEM blocks — no tag or id selectors
  • No CSS outside of blocks
  • Independent blocks -> no global reset
  • Avoid cascade
